# Ticketground 백엔드 기능 설명

이 문서는 현재 Ticketground MVP에서 백엔드 파일이 담당하는 기능을 정리한다. 프론트엔드는 별도 작업자가 진행하므로, 여기서는 API, 데이터 정책, 테스트 표면만 설명한다.

## `server.js`

Node.js 기본 `http` 모듈로 사용자 페이지와 관리자 페이지 서버를 부팅한다. 포트, 정적 파일 경로, 런타임 유틸, JSON DB 저장소, 도메인 백엔드 모듈을 조립하고 실제 요청 처리는 `backend/api-router.js`, `backend/http-handler.js`, 각 도메인 모듈에 위임한다.

`GET /api/state`는 사용자 페이지 공개 상태만 반환한다. 입장 QR의 `nonce`, `signature`, QR 메타데이터, 티켓 `ownerId`, 내부 `admissionCredentials`, 관심 공연 원장, 알림 큐, 문의 내역 같은 사용자 추적 가능 데이터는 공개 응답에서 키 자체를 제외한다. 사용자 목록도 공개 페이지에서는 `id`, `name`만 내려주며 잔액, 신뢰 점수, 제재 이력은 관리자 포트에서만 조회한다.

## `backend/runtime.js`

백엔드 공통 런타임 유틸을 제공한다. 현재 시각 고정, HMAC/해시, ID 생성, 금액 정규화, 결제수단 해석, 사용자 조회, 앱 attestation 검증을 담당한다. `TIG_APP_ATTESTATION_SECRET`이 없으면 프로세스마다 임시 랜덤값을 사용해 개발 기본값이 고정 서명으로 악용되지 않도록 한다.

## `backend/persistence.js`

로컬 JSON DB 저장소와 감사 원장 해시 체인을 담당한다.

- `loadDb`: DB 파일이 없으면 seed 데이터를 생성하고, 기존 DB는 필요한 컬렉션을 정규화한다.
- `saveDb`: JSON DB를 저장한다.
- `appendLedger`: 주요 거래/운영 이벤트를 append-only 원장에 기록한다.
- `verifyLedger`: 원장 `previousHash`와 각 entry hash를 검증한다.

## `backend/api-router.js`

`/api/*` 요청을 도메인 함수로 라우팅한다. 사용자 포트와 관리자 포트의 접근 범위를 분리하고, 사용자 포트에서는 `/api/admin/*`, `/api/ledger` 상세 조회를 노출하지 않는다.

## `backend/http-handler.js`

정적 파일 서빙과 API 응답 처리를 담당한다. 사용자 페이지는 `public`, 관리자 페이지는 `admin` 디렉터리를 기준으로 제공하며, API 오류는 JSON 형태로 표준화한다.

## `backend/dtos.js`

사용자 페이지로 내려가는 공개 DTO와 관리자 DTO를 분리한다. 공개 응답에서는 소유자 식별자, QR 내부값, 입장권 권한, 잔액, 신뢰 점수, 제재 이력 같은 민감한 필드를 제거한다.

## `backend/admission.js`

입장권 권한과 QR 보안 정책을 담당한다.

- 구매 직후 내부 `admissionCredential`을 생성하거나 갱신한다.
- 웹/앱에서 확인 가능한 가상 티켓 QR을 발급한다.
- 전용앱에서 OS 생체인증 또는 기기 잠금 인증 성공 후 신뢰 기기를 등록한다.
- 신뢰 기기 등록은 `biometricVerified: true`만으로는 불가능하고, 서버가 검증할 수 있는 앱 attestation 서명이 있어야 한다.
- 실제 입장 QR은 `APP` 채널, 신뢰 기기 토큰, 앱 attestation 서명이 모두 있어야 발급한다.
- 요청 본문의 `emergencyOverride`는 공개 API에서 효력이 없고, 서버 내부에서 `allowEmergencyOverride` 옵션을 명시한 경우에만 예외 처리할 수 있다.
- 입장 QR은 공연 약 3시간 전부터 활성화하고 20초 TTL을 적용한다.
- 게이트 검증 성공 시 티켓과 입장권 권한을 1회 사용 처리한다.

## `backend/admission-devices.js`

전용앱 신뢰 기기 등록과 기기 토큰 검증을 담당한다. OS 생체인증 또는 기기 잠금 인증 결과와 앱 attestation 서명을 모두 확인한 뒤 서버에는 기기 토큰 해시만 저장한다.

## `backend/admission-qr.js`

가상 티켓 QR, 실제 입장 QR 발급, 게이트 검증을 담당한다. 실제 입장 QR은 앱 전용 채널, 신뢰 기기, 앱 attestation, 공연 임박 활성화 조건을 만족해야 하며 20초 TTL과 1회 사용 정책을 적용한다.

## `backend/catalog.js`

공연, 회차, 좌석, 판매 상태의 기본 도메인 규칙을 담당한다. seed DB 생성, 공연별 좌석 보강, 예매 가능 여부, 공연 회차/구역 조회, 판매 요약 생성을 제공한다.

## `backend/catalog-data.js`

MVP 기본 공연장, 공연, 좌석 등급, 좌석 번호 seed 데이터를 정의한다.

## `backend/catalog-persistence.js`

기존 JSON DB가 과거 스키마여도 필요한 컬렉션과 필드를 채워 넣는 정규화 로직을 담당한다.

## `backend/seat-layouts.js`

공연장별 좌석도 메타데이터를 좌석도 API 형식으로 변환한다.

## `backend/admin-seatmaps.js`

관리자 페이지에서 공연장과 좌석도 API 연결 정보를 조회하거나 공연 개최 장소를 바꿀 때 사용하는 좌석도 관리 로직을 담당한다.

## `backend/commerce.js`

1차 예매, 결제 시뮬레이션, 공식 재판매 풀을 담당한다.

- `buyPrimary`: 판매중 좌석을 구매 처리하고 가상 티켓 및 내부 입장권 권한을 생성한다.
- `listForResale`: 보유 티켓을 정책 가격 범위 안에서 공식 재판매 풀에 등록한다.
- `joinPool`: 구매자가 특정 판매자/좌석 지정 없이 재판매 풀에 대기한다.
- `drawPool`: 조건부 랜덤 매칭을 실행하고 결제수단, 양도 수수료, 정산 정보를 기록한다.
- `directTransferAttempt`: 플랫폼 외 지정 양도 시도를 차단하고 신뢰 점수/제재 이력/감사 원장에 기록한다.

사용자 포트의 구매, 재판매, 지정 양도 차단 응답은 공개 DTO로 변환한다. 내부 `admissionCredential`, 전체 회원 객체, 결제 원장 객체, 티켓 소유자 식별자는 그대로 반환하지 않는다.

## `backend/engagement.js`

관심 공연, 예매 알림, 고객센터 문의를 담당한다.

- `userWatchlist`: 사용자별 관심 공연과 예약된 알림 작업을 조회한다.
- `upsertWatchlist`: 관심 공연 및 캘린더/알림 설정을 등록하거나 수정한다.
- `notifyWatchlist`: 상태 변경 또는 예매 알림 발송 예약/기록을 생성한다.
- `createSupportThread`, `addSupportMessage`, `updateSupportStatus`: 1:1 문의와 관리자 답변 상태를 관리한다.
- 문의/답변 발생 시 실제 카카오톡/문자 연동 전까지 `operatorAlerts`에 발송 placeholder를 쌓는다.

## `backend/session.js`

데모 환경의 로그인 상태 조회와 회원정보 수정을 담당한다. 실제 운영 인증 시스템이 아니라 사용자 페이지 기능 테스트를 위한 공개 데모 세션 API다.

- `demoSession`: 공개 사용자 목록에는 노출하지 않는 잔액, 계정 상태, 신뢰 점수를 현재 로그인 사용자에게만 내려준다.
- `updateDemoProfile`: 닉네임을 1자 이상 12자 이하로 검증한 뒤 사용자 이름을 수정하고 감사 원장에 기록한다.

### 서버 실행 및 데이터 저장

- `PORT`: 사용자 페이지 및 사용자 API 포트. 기본값은 `4173`.
- `ADMIN_PORT`: 관리자 페이지 및 관리자 API 포트. 기본값은 `50084`.
- `TIG_DB_PATH`: 테스트나 로컬 검증용 JSON DB 경로를 지정한다. 지정하지 않으면 `data/db.json`을 사용한다.
- `TIG_NOW`: 테스트에서 현재 시각을 고정할 때 사용한다. 지정하지 않으면 실제 현재 시각을 사용한다.
- `TIG_SECRET`: QR과 기기 토큰 서명에 쓰는 HMAC 비밀값이다. 감사 원장은 별도 비밀값 없이 entry hash와 `previousHash` 체인으로 변조 여부를 검증한다.
- `TIG_APP_ATTESTATION_SECRET`: 전용앱 attestation 시뮬레이션에 쓰는 HMAC 비밀값이다. 지정하지 않으면 서버 프로세스마다 임시 랜덤값을 사용하므로 앱 연동 테스트 시에는 반드시 명시해야 한다. 운영 전환 시 Apple App Attest, DeviceCheck, Google Play Integrity 같은 실제 앱/기기 검증으로 교체한다.

### 로컬 JSON DB

`loadDb`, `saveDb`, `normalizeDb`, `seedDb`가 로컬 JSON DB를 관리한다.

주요 컬렉션:

- `users`: 회원, 잔액 옵션, 계정 상태, 신뢰 점수, 제재 이력
- `venues`: 공연장과 좌석도 메타데이터
- `events`: 공연/뮤지컬/스포츠/페스티벌 판매 정보
- `tickets`: 좌석별 티켓, 소유자, 상태, 가격, 양도 횟수
- `resalePools`: 공식 재판매 풀과 조건부 랜덤 매칭 상태
- `watchlist`: 사용자 관심 공연 및 캘린더 등록 상태
- `notificationJobs`: 예매 오픈 D-3, 당일, 상태 변경 알림 예약/발송 기록
- `supportThreads`: 사용자 문의와 관리자 답변
- `operatorAlerts`: 카카오톡/문자 상담 알림을 붙이기 전까지 쓰는 운영자 알림 placeholder
- `admissionCredentials`: 구매 후 생성되는 내부 입장권 권한
- `trustedDevices`: 전용앱 신뢰 기기 등록 정보
- `qrIssueLogs`: 입장 QR 발급 로그
- `paymentTransactions`: 1차 예매 및 공식 재판매 결제 시뮬레이션 기록
- `ledger`: 주요 이벤트 해시 체인 감사 원장

### 예매 및 결제 API

- `GET /api/state`: 사용자 페이지에 필요한 공개 상태를 내려준다.
  - 사용자 목록은 `id`, `name`만 포함한다.
  - 잔액, 계정 상태, 신뢰 점수는 로그인 후 데모 세션 API에서만 확인한다.
- `GET /api/users/:userId/session`: 데모 로그인 사용자의 세션 표시 정보를 조회한다.
  - `id`, `name`, `balance`, `status`, `trustScore`를 반환한다.
  - 실제 인증 쿠키나 비밀번호 검증은 포함하지 않는 MVP 데모용 API다.
- `POST /api/users/:userId/profile`: 데모 사용자의 표시 닉네임을 수정한다.
  - 닉네임은 1자 이상 12자 이하만 허용한다.
  - 변경 내역은 `DEMO_PROFILE_UPDATED` 감사 원장으로 기록한다.
- `GET /api/users/:userId/tickets`: 특정 사용자의 보유 티켓만 공개 DTO로 반환한다.
  - `ownerId`, 입장권 권한, QR 내부값은 반환하지 않는다.
  - 사용자 페이지의 My 예매내역과 공식 재판매 등록 폼이 이 API를 사용한다.
- `POST /api/tickets/buy`: 1차 티켓 구매를 처리한다.
  - 판매중 티켓만 구매할 수 있다.
  - 판매 상태가 `ON_SALE`이 아닌 공연은 구매할 수 없다.
  - 구매 직후 가상 티켓 상태와 내부 `admissionCredential`을 생성한다.
  - 카드, 계좌이체, 무통장, 휴대폰 결제는 PG 승인 완료 또는 입금대기 상태로 시뮬레이션한다.
  - 충전금 결제는 잔액을 확인하고 차감한다.

### 관심 공연 및 알림 API

- `GET /api/users/:userId/watchlist`: 사용자의 관심 공연과 관련 알림 작업을 조회한다.
- `POST /api/watchlist`: 관심 공연을 등록하거나 수정한다.
- `POST /api/watchlist/notify`: 상태 변경, 예매 오픈 등 알림 발송 또는 예약 기록을 생성한다.

현재 알림은 실제 카카오톡/문자 발송이 아니라 `notificationJobs`와 감사 원장에 기록하는 구조다. 실제 연동 시 이 큐를 카카오 알림톡, 문자, 앱 푸시 발송기로 연결한다.

### 가상 티켓, 신뢰 기기, 입장 QR API

- `POST /api/tickets/virtual-qr`: 구매 확인용 가상 티켓 QR 정보를 발급한다.
  - 웹과 앱에서 표시 가능하다.
  - 실제 입장에는 사용할 수 없다.
  - 소유자 식별자와 내부 서명값은 응답하지 않는다.
  - 입장 QR 준비 시각과 활성화 시각을 함께 내려준다.
- `POST /api/devices/trust`: 전용앱에서 OS 생체인증 또는 기기 잠금 인증 성공 후 신뢰 기기를 등록한다.
  - 플랫폼은 생체정보 원본을 저장하지 않는다.
  - 요청자는 앱 attestation 서명을 함께 제출해야 한다.
  - 서버는 `deviceToken`과 해시된 토큰만 검증에 사용한다.
- `POST /api/tickets/qr`: 실제 입장 QR을 발급한다.
  - 기본 채널은 전용앱 `APP`만 허용한다.
  - 웹 채널은 거부한다.
  - 앱 attestation 서명과 신뢰 기기 토큰이 모두 필요하다.
  - 공연 약 3시간 전부터만 활성화된다.
  - 위험 계정은 추가 인증이 필요하다.
  - QR 토큰은 20초 TTL을 가진다.
- `POST /api/gate/verify`: 현장 게이트 검증을 처리한다.
  - 최초 1회 검증만 `valid: true`를 반환한다.
  - 검증 성공 후 티켓과 입장권 권한은 사용 처리된다.
  - 같은 QR 재사용은 `valid: false`를 반환한다.

### 공식 재판매 API

- `POST /api/resale/list`: 보유 티켓을 공식 재판매 풀에 등록한다.
  - 티켓별 최소/최대 가격 정책을 벗어나면 거부한다.
  - 공개 응답은 내부 대기자 배열 `buyers`를 그대로 노출하지 않고, 사용자 페이지 표시용 `buyerCount`만 반환한다.
- `POST /api/resale/join`: 구매자가 특정 판매자/좌석 지정 없이 재판매 풀에 대기한다.
- `POST /api/resale/draw`: 조건부 랜덤 매칭을 실행한다.
  - 초기/베타 정책에 맞춰 카드, 간편결제, 계좌이체 등 일반 PG 결제를 기본으로 시뮬레이션한다.
  - 충전금 결제를 선택한 경우에만 구매자 잔액을 확인한다.
  - 공식 양도 수수료는 거래 금액의 5%로 적용한다.
  - 구매자 최종 결제액, 양도자 정산액, 공식 양도 수수료와 결제수단을 감사 원장에 기록한다.
  - 충전금 결제 구매자의 잔액이 부족하면 매칭을 완료하지 않고 `skipped: true`, `reason: "INSUFFICIENT_BALANCE"`를 반환한다. 이때 풀은 `OPEN` 상태로 유지하고 해당 구매자만 대기열에서 제거해 재매칭할 수 있게 한다.

### 고객센터 API

- `GET /api/support/threads?userId=...`: 사용자 문의 내역을 조회한다.
- `POST /api/support/threads`: 사용자가 1:1 문의를 생성한다.
- `POST /api/support/messages`: 사용자가 기존 문의에 메시지를 추가한다.
- `POST /api/admin/support/messages`: 관리자가 답변을 남긴다.
- `POST /api/admin/support/status`: 문의 상태를 변경한다.

문의가 들어오거나 답변이 등록되면 `operatorAlerts`에 카카오톡/문자 연동용 placeholder가 생성된다.

### 관리자 API

- `GET /api/admin/summary`: 판매 현황, 재판매 풀, 고객 문의, 계정 위험도, 입장권 권한, 알림 작업, 감사 원장 상태를 조회한다.
- `GET /api/admin/venues`: 공연장과 좌석도 API 연결 정보를 조회한다.
- `POST /api/admin/events/venue`: 공연 개최 장소와 좌석도 연결을 수정한다.
- `POST /api/admin/events/sale`: 판매 정보, 판매 상태, 할인 예정, 오픈 예정, 구역별 가격을 저장한다.
- `POST /api/admin/users/statuses`: 계정 상태를 일괄 수정한다.
- `POST /api/admin/tickets/status`: 미판매 좌석의 운영 상태를 변경한다.

`/api/admin/*` 경로는 관리자 포트에서만 접근할 수 있다. 사용자 포트에서는 동일 경로를 `NOT_FOUND`로 거부해 사용자, 문의, 알림, 입장권 권한 같은 운영 데이터를 공개하지 않는다.

### 좌석도 API

- `GET /api/seat-map`: 행사 또는 공연장 기준 좌석도 데이터를 반환한다.
- `GET /api/events/:eventId/seat-map`: 특정 행사에 연결된 공연장 좌석도와 좌석 데이터를 반환한다.

잠실 올림픽 주 경기장, KSPO Dome, 블루스퀘어, 난지한강공원 기본 좌석도 메타데이터를 제공한다.

사용자 페이지의 예매 흐름은 `예매 날짜 선택 -> 예매하기 -> 좌석 선택 -> 결제` 순서다. `예매하기` 버튼은 외부 관리자 화면이나 관리자 API를 사용하지 않고 사용자 페이지 내부 좌석 선택 단계로 이동한다. 좌석도 API의 좌석 배열이 비어 있거나 티켓 좌석과 매칭되지 않는 경우에도, 현재 공연/회차의 판매 가능 티켓을 기반으로 fallback 좌석 버튼을 만들어 재판매 기능 테스트에 필요한 좌석 구매를 진행할 수 있게 했다.

### 감사 원장

- `GET /api/ledger`: 최근 감사 원장 기록을 조회한다.
- `GET /api/ledger/verify`: 해시 체인 무결성을 검증한다.

감사 원장에는 구매, 재판매, QR 발급/검증, 관심 공연, 고객 문의, 계정 상태 변경, 좌석/판매 정보 변경이 append-only 형태로 기록된다.

`/api/ledger` 상세 조회는 관리자 포트에서만 허용한다. 사용자 포트에서는 `/api/ledger/verify`의 무결성 결과만 확인할 수 있다.

## `data/db.json`

기본 로컬 실행에서 사용하는 JSON DB다. 테스트는 이 파일을 직접 변경하지 않고 임시 DB를 사용한다.

실서비스 전환 시에는 이 구조를 PostgreSQL 같은 RDBMS로 옮기고, `ledger`, `paymentTransactions`, `admissionCredentials`, `trustedDevices`, `notificationJobs`는 별도 테이블로 분리하는 것이 적합하다.
